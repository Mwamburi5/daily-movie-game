## ScoreRace

**Zone**: `DuelGame.tsx:1268–1303` — the header's right-hand column. Two
rounded score pills ("You N" / "CPU N") side by side, a turn-status caption
line below them, and a "show ends at TARGET_SCORE" hint line beneath that
(hidden once `gameOver`). Pure markup, no motion — only opacity/ring
transitions via Tailwind, no framer-motion.

**Props in**:
- `playerScore: number` — `useState`, mutated in ~9 places (player plays,
  lay-offs, melds, Final Cut).
- `cpuScore: number` — `useState`, mutated symmetrically for CPU actions.
- `status: DuelStatus` — `useState`, drives which pill gets the active ring
  (`'playerTurn'` → player pill; `'cpuTurn' | 'recastOffer'` → CPU pill).
- `runState: RunState | null` — `useState`; when non-null on the player's
  turn the caption reads `Run ×${runState.count + 1}?` instead of "Your
  turn".
- `gameOver: boolean` — derived (`status === 'over'`), gates the "show ends
  at…" hint.
- `TARGET_SCORE: number` — imported constant from `lib/duel.ts` (race-to-20).

**Callbacks out**: none. Purely presentational — reads state, emits nothing.

**Behavior notes**: no `AnimatePresence`/layout animation; `data-score`,
`data-turn`, `data-target-hint` attributes are relied on by the sim-adjacent
browser checks (do not drop them from a replacement). z-index: sits in the
plain document flow of `<header>`, not absolutely positioned, so no
viewport-edge pinning to preserve. Not difficulty-gated.

**Risk**: low. Fully self-contained given the 4 state values above; no
shared refs. The only subtlety is that `status === 'recastOffer'` must keep
the CPU pill "active" — easy to drop by accident if a future component only
checks `'cpuTurn'`.

## TazCorner

**Zone**: `DuelGame.tsx:1306–1341` — the CPU presence block, absolutely
positioned (`absolute inset-x-0 top-[64px] z-10`) directly under the header,
pinned to the top of the board (not a safe-area edge, but a fixed pixel
offset from the top — see Risk). Contains: a fanned row of face-down card
backs (`motion.div` per card, `layoutId={id}` for FLIP-style reflow as the
hand shrinks, `-ml-4` overlap), a "CPU · N cards" label, and two token chips
(Final Cut / Recast) rendered inline (not the shared `TokenChips` component
— see TokenChips section for the player-side twin markup elsewhere).

**Props in**:
- `cpuHand: string[]` — `useState`, card ids only (never revealed — always
  rendered as a plain "M" glyph, no `CardView`/face art).
- `cpuTokens: Tokens` — `useState<{finalCut: boolean; recast: boolean}>`,
  toggled off when spent.

**Callbacks out**: none. Nameplate has no interactive affordance — it's
inert display; taps on the card backs do nothing (no `onTap`/`onClick`
wired here, unlike the player's pile-top which has `onTap={flipCard}`).

**Behavior notes**: `motion.div layoutId={id}` per card back means Framer
tracks each id across renders/removals for a smooth collapse as `cpuHand`
shrinks; a replacement component must keep stable `key`/`layoutId` bound to
the card id, not array index, or the collapse animation breaks. No
`AnimatePresence` (no exit animation defined — cards vanish immediately when
removed from the array, which is the current, presumably-acceptable,
behavior; note but don't "fix"). Token label strings are literal ("Final
Cut", "Recast") — not difficulty-gated. Not gated by any `Difficulty` value.

**Risk**: medium. The `top-[64px]` offset is a magic number coordinated with
the header's actual rendered height — if a redesign changes header height
this zone's offset must move in lockstep (currently not derived from a
layout measurement, just hardcoded like every other zone in this file — see
general Risk note on absolute stacking in the "Cross-zone" section below).

## MeldShelf

**Zone**: primarily the standalone `src/components/MeldZone.tsx` (59 lines,
full file), mounted at `DuelGame.tsx:1556–1563`. Renders one row per banked
`Meld`, each a horizontal strip of mini card swatches (`posterColor`
background rectangles, or a gold ★ tile for wild ids) with a truncated label
(`meldLabel(meld)  ×N`). The whole zone is `absolute inset-x-0 top-[440px]
z-10` (see MeldZone.tsx:24) and returns `null` entirely when `melds.length
=== 0` (no empty-state shelf).

**Props in** (already a real component — this is its EXISTING contract, not
one to invent):
- `melds: Meld[]` — from `DuelGame.tsx`'s `useState<Meld[]>([])`, appended by
  both `bankMeld()` (player) and the CPU turn effect's meld branch
  (`DuelGame.tsx:914`), and mutated in place (via `.map`, not append) by both
  `playerPlay`'s lay-off branch (`:428`) and the CPU's `doLayoff`
  (`:945`) — **melds are written to from at least 4 call sites**, not just
  one owner.
- `highlightIds: ReadonlySet<number>` — derived in `DuelGame.tsx` as
  `meldHighlights` (`:304`), itself a union of `layOffTargets` (raised-card
  lay-off eligibility) and `hintMeldId` (an active hint's target row).
- `setRowRef: (id: number, el: HTMLDivElement | null) => void` — callback
  that lets `DuelGame.tsx` populate `meldRowRefs` (a `Map<number,
  HTMLDivElement>` ref), used by `findMeldAt()` (`:345`) to hit-test a
  dropped card's screen coordinates against banked meld rows during drag/drop
  lay-off resolution.

**Callbacks out**: only `setRowRef` — invoked once per rendered row via
`ref={(el) => setRowRef(meld.id, el)}`. This is NOT a click handler; it's DOM
geometry plumbing the parent needs for pointer-based lay-off drops.

**Behavior notes**: `motion.div layoutId={cid}` per real card swatch (wilds
render as a plain non-motion `div`) — so a card animates (FLIPs) from
hand/pile into the shelf when melded or laid off, sharing a `layoutId`
namespace with `Hand.tsx` and the pile-top `motion.div`s in `DuelGame.tsx`.
**A MeldShelf replacement must preserve the shared `layoutId={cardId}`
convention across all three surfaces (hand, pile, meld shelf) or the
fly-to-shelf animation breaks.** `highlightIds.has(meld.id)` toggles a
`ring-2 ring-[#7a5a10]` — this is the drop-target/hint affordance, not
decorative. Not difficulty-gated.

**Risk**: high (the plan doc's own §4 flags this as needing "growth
strategy" design work). The `top-[440px]` pin sits below the pile/take-glow
zone (`top-[160px]` + card heights) and above the hand — any relayout must
keep it clear of the turn banner (`top-[398px]`) and the idle cue
(`top-[372px]`), which currently visually stack in the same mid-board band
by careful pixel coordination, not a layout system. `meldRowRefs` is
imperative DOM-geometry state owned by `DuelGame.tsx` and MUST be re-wired
identically by any replacement — dropping `setRowRef` silently breaks
lay-off drop detection (no compile error, only a runtime hit-test miss).

## RecastOffer

**FLAG — name mapping is inferred, not attested.** "Recast" does not appear
as a component name or code comment describing an overlay in
`DuelGame.tsx`; the term is confirmed elsewhere in the repo (memory: "Recast
offer overlay (the drama moment)" in `docs/orchestration-plan.md:78`, and
`RecastOffer` interface at `DuelGame.tsx:92`) as the **CPU super-link /
Final Cut suspense overlay** — a DIFFERENT mechanic from the "take-to-meld /
TAKE TO FINISH MELD" glow the task brief also gestures at. These are two
distinct existing mechanics; documenting both so the orchestrator can
confirm which (or both) "RecastOffer" the forge component should target.

### Candidate A (matches the `RecastOffer` type name in code) — CPU offer overlay

**Zone**: `DuelGame.tsx:1824–1866`. A full-board modal
(`absolute inset-0 z-[90]`, dim backdrop `bg-[#23211c]/35`), wrapped in
`AnimatePresence` with fade in/out (`initial/animate/exit opacity`,
`duration: reduce ? 0.1 : 0.2`). Shows the CPU's held-in-suspense play: a
title card, "Final Cut — no connection needed" or "Super link — +4 and an
encore" subtext, a `CardView` of the offered movie, and two full-width
buttons: "Recast — cancel it" and "Allow it".

**Props in**:
- `recastOffer: RecastOffer | null` — `useState`; shape `{ id: string, drew:
  boolean, finalCut: boolean, pileIdx: number }` (`DuelGame.tsx:92–97`), set
  by `cpuAttempt()` (`:827`) whenever the CPU's move is a super link or Final
  Cut AND the player still holds a `recast` token.
- `offerMovie: Movie | null` — derived (`mv(recastOffer.id)`), `:1216`.
- `reduce: boolean` — `useReducedMotion()` result, threaded through for the
  fade timing only.

**Callbacks out**:
- `playerRecast: () => void` (`:843`) — spends the player's recast token,
  cancels the CPU's play, returns a drawn-but-unresolved card to the CPU's
  hand if `recastOffer.drew`, resets `cpuRun`, sets `status: 'playerTurn'`.
- `allowCpuPlay: () => void` (`:837`) — resolves the held play via
  `resolveCpuPlay(recastOffer)`.
- Both are guarded by `resolvingOffer` (a `useRef<boolean>`, not state) to
  block a double-tap from resolving the same offer twice — **this ref MUST
  be threaded through or owned by whatever replaces this zone**; it is not
  optional plumbing.

**Behavior notes**: `status` is forced to `'recastOffer'` while this is up
(`cpuAttempt`, `:831`) — ScoreRace's CPU-pill-active check and the turn
caption both key off this status value, so RecastOffer's mount/unmount is
coupled to that tri-state `DuelStatus` more broadly than just its own
overlay. Not difficulty-gated (every tier's CPU can trigger it; only
`mayRecast`/`k` difficulty knobs decide whether the CPU itself would ever
recast the PLAYER — see `cpuWouldRecast`, `:342`, a separate, asymmetric
mechanic not shown to the player at all).

### Candidate B — take-to-meld / "TAKE TO FINISH MELD" glow

**Zone**: `DuelGame.tsx:1438–1484`, nested inside the Double-Feature pile
render loop (`:1379–1489`). Not a modal — an in-place glow + button
overlaid on a pile-top card. Two visual modes gated by `takeGlowEnabled`
(`difficulty === 'matinee'` only, `:186`):
  - **Matinee** (loud): animated gold glow ring (`motion.div`, `opacity:
    [0.45, 1, 0.45]`, `repeat: Infinity`, skipped entirely if `reduce`) plus
    a big gold pill button reading "↑ Take to finish meld".
  - **Feature/Director's Cut** (subtle): same tap target, no glow, a small
    dark-green pill reading "↑ Take for meld". No motion at all.

**Props in**:
- `takeTargets: (Movie | null)[]` — derived per-pile (`:294`), the union of
  `takeTop(pileIdx)` (auto take-to-meld — hand already holds 2 the top would
  complete) and `meldSelectTake(pileIdx)` (the Meld-mode 2-card selection
  discovery path, Matinee-only per `takeGlowEnabled` guard at `:282`).
- `takeGlowEnabled: boolean` — `difficulty === 'matinee'` (`:186`).
- `reduce: boolean` — suppresses the glow motion only, not the button.

**Callbacks out**:
- `doTakePile(pileIdx: number): void` (`:591`) — lifts the pile top into the
  player's hand (does NOT bank the meld immediately; banks next turn), with
  pile reseed-from-deck logic if it empties the pile to length 1.

**Behavior notes**: this is a per-pile decoration, not a standalone overlay
— it lives INSIDE the pile markup (shares the `pileZoneRefs` drop-zone div).
A forge extraction would need to either (a) extract just the glow+button as
a child component taking `{ target: Movie | null; glowing: boolean; reduce:
boolean; onTake: () => void }`, mounted twice (once per pile) inside the
existing pile markup — leaving pile rendering itself untouched — or (b) be
descoped from "RecastOffer" entirely since it shares no code, state, or
visual language with Candidate A.

**Risk**: the ambiguity itself is the risk. If the orchestrator intends
Candidate A, this section's contract is clean (one modal, 2 callbacks, 1
ref-guard). If Candidate B, it is NOT a standalone-overlay shape at all —
it's two small injected decorations inside a zone (`ScoreRace`-adjacent pile
markup) that isn't itself slated for a forge component in the plan's
candidate list. Recommend the orchestrator confirm which is meant before
briefing a `RecastOffer.tsx` forge task; do not guess further.

## DrawChoice

**Zone**: `DuelGame.tsx:1790–1822`. A full-board modal (`absolute inset-0
z-[85]`, semi-opaque backdrop, NOT wrapped in `AnimatePresence` — it mounts/
unmounts immediately with no enter/exit transition, unlike RecastOffer and
the game-over screen). Shows a "Keep one — the other two leave play"
caption, then the (up to 3) revealed cards face-down (`CardView faceUp=
{false}`) each optionally tagged "Connects" if it shares a person with
either pile top, then a "Tap to keep…" footer line.

**Props in**:
- `drawChoice: string[] | null` — `useState<string[] | null>(null)`
  (`:205`), set to `deck.slice(0, 3)` by `playerDraw()` (`:568`); `null`
  means "not choosing."
- `tops: Movie[]` — derived per-pile linking tops (`:247`), used per-card to
  compute the "Connects" badge (`sharedPeople(t, mv(id)).length > 0`).

**Callbacks out**:
- `playerPickDraw(id: string): void` (`:574`) — clears `drawChoice`, adds
  the picked id to `playerHand`, sets it as `pendingDraw` + `raisedId` (so it
  immediately enters the keep/toss/play flow), and emits a `say()` banner
  ("kept it — it connects!" or "kept the card").

**Behavior notes**: the other 1–2 unpicked cards are simply never referenced
again — they were already spliced out of `deck` at draw time
(`playerDraw`, `:567`) and are NOT tracked anywhere (per decision D1: burned
cards are invisible, not shown discarding). A DrawChoice replacement must
not try to animate the "losing" cards away since there is no state
representing where they went — they're already gone from `deck` before this
modal even renders. z-index `z-[85]` sits below RecastOffer (`z-[90]`) and
the game-over screen (`z-[100]`) but above the turn banner (`z-40`) and
meld-select bar (`z-[55]`) — respect this stacking order if reused. Not
difficulty-gated.

**Risk**: low. Clean, fully self-contained given `drawChoice` + `tops` +
one callback. The one subtlety: this modal and `pendingDraw`'s keep/toss bar
(`:1588–1615`) are mutually exclusive by construction (`playerPickDraw`
clears `drawChoice` and sets `pendingDraw` in the same call) but a naive
extraction that duplicates this logic elsewhere could violate that
invariant — the sequencing must stay atomic.

## PlayBanner

**Zone**: `DuelGame.tsx:1491–1538`. A pill centered at a fixed board
position (`absolute inset-x-0 top-[398px] z-40`, `pointer-events-none` on
the wrapper so it never blocks taps underneath), `AnimatePresence`-wrapped
with a spring-in / fade-out transition (`type: 'spring', stiffness: 320,
damping: 24`, or `duration: 0.15` under `reduce`). Shows a who-chip ("You"/
"CPU"), the lowercase `say()` message text, an optional "Deep cut" teal
badge, and an optional `+N` points badge. Background color varies by
`banner.tier` (`super` → burnt orange, `strong` → dark gold, else near-
black).

**Props in**:
- `banner: DuelBanner | null` — `useState<DuelBanner | null>(null)`
  (`:225`), shape `{ who: 'You'|'CPU', text: string, tier: LinkTier|null,
  points: number|null, deep: boolean, seq: number }` (`:107–114`). Set
  exclusively via the `say()` helper (`:307–313`), called from ~20 sites
  across both player and CPU turn logic — this is the most heavily
  multi-writer piece of state in the file.
- `reduce: boolean`.

**Callbacks out**: none — read-only display. Auto-dismiss is handled by a
`useEffect` in `DuelGame.tsx` (`:1122–1126`, 2400ms timeout keyed on
`banner`), NOT by the zone itself — a forge `PlayBanner` component could
either stay purely presentational (parent keeps owning the dismiss timer,
current shape) or take on the timer itself if given an `onDismiss` prop;
brief should say which.

**Behavior notes**: `key={banner.seq}` on the `AnimatePresence` child is
load-bearing — `seq` is a monotonic counter (`useRef`, `:238`) incremented
on every `say()` call specifically so that two consecutive banners with
identical text (e.g. two "kept the card" in a row) still remount and
retrigger the enter animation instead of no-opping as "the same node."
**Any replacement must preserve a monotonic remount key, not just diff on
banner content.** Not difficulty-gated. Sits below `DrawChoice` (`z-[85]`)
and above the deck/piles (`z-10`), `MeldShelf` (`z-10`), and the idle cue
(`z-10`) — it's meant to float over the mid-board.

**Risk**: low-medium. Multi-writer (`say()` from ~20 call sites) is a
feature here, not a bug — the banner IS the shared narration channel; a
forge component just needs to accept the `DuelBanner` shape faithfully. The
only real risk is the `seq`-as-remount-key convention being dropped by an
agent who "simplifies" the `AnimatePresence` key to `banner.text`.

- `handHasMeld: boolean` — derived (`bestMeld(playerHandMovies) !== null`,

- Final Cut chip `onClick`: inline arrow at `:1675–1679` — toggles

That's a reasonable design call but IS a design call, not a given — the
current code doesn't share any markup or logic between the two today; note

## IdleCue

**Zone**: `DuelGame.tsx:1540–1553`. A single pill, `absolute inset-x-0
top-[372px] z-10`, `pointer-events-none`, reading "one move — play a card or
draw." No motion (no framer-motion wrapper at all — plain conditional JSX).

**Props in** — the entire zone is one boolean expression evaluated inline,
not stored as derived state:
```
status === 'playerTurn' &&
!runState &&
pendingDraw === null &&
drawChoice === null &&
!meldSelect &&
raisedId === null &&
!gameOver
```
A forge component would need all six of `status`, `runState`, `pendingDraw`,
`drawChoice`, `meldSelect`, `raisedId`, plus `gameOver`, OR the parent
precomputes this boolean and passes a single `visible: boolean` prop
(recommended — cheap, avoids duplicating the six-condition guard in two
places).

**Callbacks out**: none.

**Behavior notes**: `top-[372px]` sits ABOVE the turn banner's `top-[398px]`
in document order but the two are positionally close (26px apart) — they
are designed to never both be visible at once (the banner appears the
moment ANY action happens, at which point at least one of the six IdleCue
conditions goes false), but nothing in the code enforces mutual exclusion
structurally; it's an emergent property of the state machine. A redesign
that changes banner timing (e.g. a longer auto-dismiss) could cause the two
to visually collide. Not difficulty-gated.

**Risk**: low. Simplest zone in the file. Only risk is silently dropping one
of the seven guard conditions when translating to a `visible` prop —
enumerate all seven in the brief so the forge agent doesn't infer a subset.

## Cross-zone notes (not a component, but load-bearing for every forge brief)

- **Absolute-pixel stacking, not a layout system.** Every zone above is
  positioned via a hardcoded `top-[Npx]` / `bottom-[Npx]` value tuned by eye
  against the `max-w-[420px]` phone frame (`DuelGame.tsx:1246`). There is no
  flex-zone or safe-area-aware layout today (per the orchestration plan
  §5 Wave A, that's a planned future HERO pass, not yet done). A forge
  component built against a fixed pixel offset will need re-pinning when
  that layout pass lands — don't treat any `top-[Npx]`/`bottom-[Npx]` value
  in this doc as permanent, only as "where it is today."
- **`layoutId` is a shared namespace across zones.** Card ids are used as
  `layoutId` in the CPU hand (TazCorner), the pile tops, `Hand.tsx` (player
  hand), and `MeldZone.tsx` (meld shelf) — Framer Motion uses this to FLIP-
  animate a single card's movement between these surfaces. Any forge
  component touching a card's `motion.div` must keep `layoutId={cardId}`
  identical in shape to what exists today, or cross-zone fly animations
  (hand → pile, pile → meld shelf, etc.) will silently stop animating (no
  error — Framer just treats them as unrelated elements).
- **`z-index` map observed today** (low → high): pile/deck/hand zones and
  MeldShelf/IdleCue (`z-10`), tap-elsewhere backdrop (`z-20`), token chips/
  Meld button/hint/sort (`z-40`), turn banner (`z-40`), keep/toss + run +
  meld-select bottom bars (`z-[55]`), super-link flash (`z-[60]`),
  draw-choice modal (`z-[85]`), recast offer modal (`z-[90]`), game-over
  screen (`z-[100]`). Preserve relative order if any zone is extracted.
- **`resolvingOffer` (a `useRef`, not `useState`) is a cross-cutting guard**
  that only RecastOffer's two callbacks touch, but it's easy to miss because
  refs don't show up in a `useState` audit. Any RecastOffer forge component
  must either receive this ref forwarded or have the guard logic stay
  entirely in `DuelGame.tsx`'s callback implementations (recommended — don't
  hand a raw mutable ref into a "dumb" presentational component).
- **`melds` (Meld[]) has 4 writers** (player bank, player lay-off, CPU bank,
  CPU lay-off) and is read by both `MeldShelf` (display) and the lay-off
  hit-testing (`findMeldAt`, uses `meldRowRefs` populated via MeldShelf's
  `setRowRef`). This is the single most shared piece of state across forge
  candidates — `MeldShelf` and any future lay-off-drop-zone logic are
  coupled through the `meldRowRefs` ref map, not just through props.
